# factory

A tiny, typed **model factory** for generating and persisting test data, in the
spirit of Laravel's model factories or `factory_boy`. You describe how a model
looks once; the factory builds fresh, [Faker](https://faker.readthedocs.io/)-powered
instances on demand — either in memory or persisted through your ORM session.

The library is **ORM agnostic**: it depends only on a small structural
`Persister` protocol (`add` / `add_all` / `commit` / `refresh`), so it works
out of the box with both [SQLModel](https://sqlmodel.tiangolo.com/) and plain
[SQLAlchemy](https://www.sqlalchemy.org/) — or anything else exposing the same
Unit-of-Work API — without importing either.

## Features

- **Declarative factories** — subclass `BaseFactory[Model]`, set `model`, and implement `defaults()`.
- **In-memory or persisted** — `make()` builds unsaved instances; `create(session)` persists them.
- **Per-instance overrides** — chainable, immutable `with_attributes(...)`.
- **Dynamic values** — any callable in `defaults()` is a callback, optionally resolved against the session (great for relationships).
- **Field projections** — `only(*fields)` returns plain dicts instead of model instances.
- **Batches** — every builder takes `count=` to produce lists.
- **Fully typed** — generic over the model type, so your IDE and type checker know what each factory returns.

## Requirements

- Python **3.14+**
- One ORM backend of your choice (SQLModel or SQLAlchemy) if you intend to persist with `create()`.

## Installation

Pick the extra that matches your ORM:

```bash
# with SQLModel
uv add "factory[sqlmodel] @ git+https://github.com/jurios/factory.git"

# with SQLAlchemy
uv add "factory[sqlalchemy] @ git+https://github.com/jurios/factory.git"
```

The core package only depends on `faker`; the ORM is an optional extra. You can
also install without any extra if you only ever call `make()` / `only()` (no
persistence).

## Quickstart

```python
from typing import Any

from faker import Faker
from sqlmodel import Field, SQLModel

from factory import BaseFactory


class User(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    name: str
    email: str
    age: int


class UserFactory(BaseFactory[User]):
    model = User

    @classmethod
    def defaults(cls, faker: Faker) -> dict[str, Any]:
        return {
            "name": faker.name(),
            "email": faker.email(),
            "age": faker.pyint(min_value=18, max_value=99),
        }


user = UserFactory().make()            # a fresh, unsaved User
users = UserFactory().make(count=3)    # a list of three Users
```

## Defining a factory

A factory is a subclass of `BaseFactory[Model]` that provides two things:

- `model`: the class to instantiate.
- `defaults(cls, faker)`: a classmethod returning the attribute dictionary used
  to build each instance. It receives a `Faker` instance for random data.

```python
class UserFactory(BaseFactory[User]):
    model = User

    @classmethod
    def defaults(cls, faker: Faker) -> dict[str, Any]:
        return {
            "name": faker.name(),
            "email": faker.email(),
            "age": faker.pyint(min_value=18, max_value=99),
        }
```

`defaults()` is called once per built instance, so every object gets its own
random values.

## API reference

All builders live on `BaseFactory[Model]`. A factory instance is cheap and
immutable — `with_attributes()` returns a new factory rather than mutating the
current one, so factories are safe to reuse and chain.

### `make(count=None) -> Model | list[Model]`

Builds instances **in memory**. Nothing is persisted and no session is needed.
Callbacks are **not** resolved: any callable value collapses to `None` (see
[Dynamic values](#dynamic-values-and-callbacks)).

```python
UserFactory().make()             # -> User
UserFactory().make(count=5)      # -> list[User] of length 5
```

### `create(session, *, count=None) -> Model | list[Model]`

Builds instances and **persists** them through `session`, resolving callbacks
along the way. For a single object it calls `add` / `commit` / `refresh`; for a
batch it calls `add_all` / `commit` and refreshes each row, so returned
instances have database-populated fields (like autogenerated ids).

```python
user = UserFactory().create(session)              # persisted, user.id is set
users = UserFactory().create(session, count=10)   # ten persisted Users
```

`session` can be any object satisfying the [`Persister`](#the-persister-protocol)
protocol — a SQLModel `Session`, a SQLAlchemy `Session`, or your own.

### `with_attributes(**kwargs) -> BaseFactory[Model]`

Returns a **new** factory with the given attributes overriding the defaults.
Calls are chainable and merge left to right (last write wins on the same key).
Overrides take precedence over defaults, including over callback defaults.

```python
UserFactory().with_attributes(name="Ada Lovelace").make()
UserFactory().with_attributes(name="Grace").with_attributes(age=36).make()
```

An override value may itself be static or a callback, and follows the same
callback rules as `defaults()`.

### `only(*fields, count=None) -> dict[str, Any] | list[dict[str, Any]]`

Builds instances in memory (like `make()`) and returns a projection containing
only the requested fields, as plain dictionaries. Useful for request payloads
or assertions. Like `make()`, it does not resolve callbacks.

```python
UserFactory().only("name", "email")
# {"name": "...", "email": "..."}

UserFactory().only("name", count=2)
# [{"name": "..."}, {"name": "..."}]
```

### `defaults(cls, faker) -> dict[str, Any]`

The one method every factory must override. Raises `NotImplementedError` on the
base class. Returns the attribute dictionary used to build each instance.

### The `Persister` protocol

`create()` accepts anything structurally compatible with this protocol, which is
all the library needs from a database session:

```python
@runtime_checkable
class Persister(Protocol):
    def add(self, instance: Any) -> Any: ...
    def add_all(self, instances: Any) -> Any: ...
    def commit(self) -> Any: ...
    def refresh(self, instance: Any) -> Any: ...
```

Because it is structural (duck typing), a SQLModel `Session` and a SQLAlchemy
`Session` both satisfy it without any adapter — SQLModel's `Session` is itself a
subclass of SQLAlchemy's. You can also implement it yourself to target a
different backend.

## Dynamic values and callbacks

Any **callable** value in `defaults()` (or in `with_attributes()`) is treated as
a *callback* rather than a literal value. Callbacks let you defer computation
until build time and, crucially, react to the persistence session.

A callback is resolved differently depending on the builder:

| Builder      | Callback behavior                                             |
| ------------ | ------------------------------------------------------------- |
| `make()`     | Not resolved — the field becomes `None`.                      |
| `only()`     | Not resolved — the field becomes `None`.                      |
| `create()`   | Resolved. The result is used as the field value.              |

When `create()` resolves a callback, it inspects the callback's signature:

- **Zero parameters** → called as `callback()`.
- **One or more parameters** → called as `callback(session)`, giving the callback
  access to the live session (for example, to create and reference related rows).

```python
class ProfileFactory(BaseFactory[Profile]):
    model = Profile

    @classmethod
    def defaults(cls, faker: Faker) -> dict[str, Any]:
        return {
            "handle": faker.user_name(),
            "token": lambda: "fixed-token",                       # no-arg callback
            "user_id": lambda session: UserFactory()              # session-aware callback
                .create(session).id,
        }
```

- `ProfileFactory().make().token` → `None` (callbacks collapse in memory).
- `ProfileFactory().create(session).token` → `"fixed-token"`.
- `ProfileFactory().create(session).user_id` → the id of a freshly created `User`.

A static override always beats a callback default:

```python
ProfileFactory().with_attributes(token="manual").create(session).token
# "manual"
```

## Examples

### Overriding attributes

```python
admin = UserFactory().with_attributes(name="Root", age=99).make()
```

### Building related objects

Use a session-aware callback to create and link related rows in a single
`create()` call:

```python
from sqlmodel import Field, Session, SQLModel, create_engine


class Profile(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    handle: str
    token: str | None = None
    user_id: int | None = None


class ProfileFactory(BaseFactory[Profile]):
    model = Profile

    @classmethod
    def defaults(cls, faker):
        return {
            "handle": faker.user_name(),
            "user_id": lambda session: UserFactory().create(session).id,
        }


engine = create_engine("sqlite:///:memory:")
SQLModel.metadata.create_all(engine)

with Session(engine) as session:
    profile = ProfileFactory().create(session)
    # profile is persisted and profile.user_id points at a new User row
```

### The same factory with plain SQLAlchemy

Nothing changes on the factory side — only the model definition and session:

```python
from typing import Any

from faker import Faker
from sqlalchemy import String, create_engine
from sqlalchemy.orm import DeclarativeBase, Mapped, Session, mapped_column

from factory import BaseFactory


class Base(DeclarativeBase):
    pass


class Account(Base):
    __tablename__ = "accounts"

    id: Mapped[int] = mapped_column(primary_key=True)
    username: Mapped[str] = mapped_column(String(50))
    email: Mapped[str] = mapped_column(String(120))


class AccountFactory(BaseFactory[Account]):
    model = Account

    @classmethod
    def defaults(cls, faker: Faker) -> dict[str, Any]:
        return {"username": faker.user_name(), "email": faker.email()}


engine = create_engine("sqlite:///:memory:")
Base.metadata.create_all(engine)

with Session(engine) as session:
    account = AccountFactory().create(session)
    accounts = AccountFactory().create(session, count=3)
```

### Deterministic data in tests

Seed Faker to make generated values reproducible:

```python
from faker import Faker

Faker.seed(0)
UserFactory().make().name  # stable across runs
```

## Development

The project uses [uv](https://docs.astral.sh/uv/). Install the toolchain and
dependencies (both ORM backends are part of the dev group):

```bash
uv sync
```

Run the test suite and the static checks:

```bash
uv run pytest              # unit tests + SQLModel and SQLAlchemy integration tests
uv run ruff check .        # lint
uv run ruff format --check .
```

The integration suites (`tests/unit/factory/test_sqlmodel_backend.py` and
`test_sqlalchemy_backend.py`) exercise `create()` against a real in-memory SQLite
database through both backends, so the ORM-agnostic behavior is verified end to
end. SQLite needs no extra system package — Python ships the `sqlite3` module
compiled in.
